# Security Review Checklist

- Canonicalization rejects floats, unknown fields, ambiguous signatures, and reordered-object drift.
- Relays validate schema, timestamp window, key activity, replay nonces, revocation state, and signatures.
- Public tables contain commitments and canonical proof objects only; raw messages stay local.
- Checkpoint settlement compares stored on-chain roots, counts, previous checkpoint, epoch, shard, and relay ID.
- Negative attestations require evidence commitment, expiry, issuer identity, and appeal metadata before production use.
- Relay, provider, auditor, and checkpoint keys use env/file only in development; production must use KMS/HSM adapters.
- KMS/HSM signing adapters must satisfy the `SigningAdapter` interface and must not expose raw Ed25519 seeds to service processes.
- ZK proving keys generated by `zk:setup` are development artifacts; production ceremonies need separate custody and reproducibility review.
- Incident response must cover key compromise, relay equivocation, database rollback, chain reorg, and malicious scoring provider.
